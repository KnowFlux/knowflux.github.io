# Day 14 — Modal Makeover & Server Port Resilience

## Goal
Two independent goals today:
1. **Omni‑Dex Modal:** Transform the dark sci‑fi modal into a bright Nintendo‑inspired KnowFlux experience.
2. **Server:** Make the backend more robust by handling port conflicts gracefully (no more "Address already in use" crashes).

---

## What We Did

### 1. Omni‑Dex Modal — Full Retro Re‑style
The modal was previously a dark glassy HUD (`#1a1a2e` background, rounded corners, soft shadows). That didn't fit the KnowFlux identity. We rewrote it to match the **bold Nintendo energy** described in the README.

**CSS changes** (`css/omni-dex.css`):
- **Overlay:** Changed from `rgba(0,0,0,0.7)` to saturated KnowFlux blue (`rgba(67,146,241,0.55)`) — more energetic
- **Modal box:** White background, **4px solid black** border, zero border-radius (square = retro), chunky **8px drop shadow** in the accent colour
- **Per‑universe shadows:** The `.omni-modal-content` now gets a `data-universe` attribute, so Exploded cards get gold shadow (`#D4A017`) and Pinnacle gets blue (`#4392F1`)
- **Titlebar:** Orange underline (`#F46036`), **Anton** font for the title, uppercase, bigger (`1.5rem`)
- **Stats button:** Blue background, black border, hard 4px shadow, Anton font — looks like a Nintendo switch button
- **Body text:** Changed from dull `#e0e0e0` on dark to `#333` on white — much more readable
- **Sidebar:** White panel, black border, orange accent shadow instead of dark
- **Tooltips:** White background, black border, orange shadow — clear and punchy
- **Control buttons:** Square (no border-radius), black border, hard shadow, Anton uppercase. Close is orange, Next is blue
- **Entrance animation:** Scale‑up pop with bouncy cubic‑bezier (`0.25s cubic-bezier(0.34, 1.56, 0.64, 1)`)
- **Titlebar padding:** Added `8px` right padding so the stats button's shadow doesn't get clipped

**JS changes** (`js/omni-dex.js`):
- Added `modalContentOuter` reference to the outer `.omni-modal-content` container
- In `openModal()`: sets `modalContentOuter.dataset.universe = entry.universe || 'pinnacle'` so the CSS can style the shadow colour per universe

### 2. Backend Server — Port Conflict Resilience
The server used to crash with "Address already in use" if the previous instance hadn't fully released the port. We added automatic port cleanup.

**Changes** (`Backend/server.py`):
- **`free_port()` function:** Uses `lsof -ti :PORT` to find any process listening on the given port and kills it with `SIGKILL`. This runs before the server starts.
- **`ReusableTCPServer` class:** Extends `socketserver.TCPServer` and overrides `server_bind()` to call `setsockopt(SOL_SOCKET, SO_REUSEADDR, 1)` on the socket before binding. This tells the OS to reuse the port immediately instead of waiting for `TIME_WAIT`.
- **Retry loop:** Wrapped the server startup in `for attempt in range(5)` — if binding fails with `errno 48` (EADDRINUSE), it waits 2 seconds and retries up to 5 times. After 5 failures, exits with code 1.
- **Default port changed** from `5000` to `5001` to avoid conflicts with common services
- **`__name__ == '__main__'` guard:** Wrapped the server startup so importing the module doesn't accidentally start the server
- Added the necessary imports: `sys`, `subprocess`, `socket`, and `time`

### 4. Modal Overlay Fix — CSS Variable Cascade Debugging
After the initial modal restyle, the Exploded universe overlay remained blue instead of turning dark red. The problem was subtle: JS set `data-universe` on the `.omni-modal` element, but the CSS only defined Exploded overlay variables on `body.omni-dex[data-universe="exploded"]` — which never matched, because JS never set `data-universe` on the body.

**Root cause:** CSS custom properties cascade from parent to child in the DOM. The `--modal-overlay` variable existed on the body, but the modal's `background` was hardcoded to `transparent` instead of `var(--modal-overlay)`. And even after fixing that, the body-level `[data-universe="exploded"]` rule wouldn't help because the attribute wasn't on the body.

**Two‑part fix** (`css/omni-dex.css`):
- **`background: transparent` → `background: var(--modal-overlay)`** on `.omni-modal` — now the overlay actually uses the variable instead of ignoring it
- **Added `.omni-modal[data-universe="exploded"]` and `.omni-modal[data-universe="pinnacle"]`** rules that override `--modal-overlay` directly on the modal element. Since JS sets `data-universe` on the modal, these selectors match and the colour changes properly

**Lessons learned:**
- **CSS variables cascade down the DOM tree** — a child element inherits variables from its ancestors, but not from siblings or unrelated branches
- **Attribute selectors only match the element they're on** — `.modal-content[data-universe]` won't match if the attribute is on the parent `.modal`
- **Hardcoded values defeat CSS variables** — `background: transparent` ignores the variable entirely, no matter what it's set to
- **Keep the attribute close to where it's used** — setting `data-universe` on the element that needs the variable avoids cascade confusion. Don't set it on `body` if only the modal cares
- **Corollary:** The `.omni-modal-content[data-universe="exploded"]` rule still has the same problem — it sets `--modal-bg`, `--modal-shadow`, etc., but `data-universe` is on the modal overlay, not the content element. That rule also doesn't work. The pattern to fix all of them: `.omni-modal[data-universe="exploded"] .omni-modal-content` instead.

This took hours to debug, but the lesson about CSS variable scope and attribute selector matching is one you'll never forget.

## Lesson Learned
**Retro ≠ outdated.** Bright white backgrounds with thick black borders and hard shadows are a conscious design choice that communicates energy, playfulness, and brand consistency. Meanwhile, server code should always anticipate failure — ports get stuck, and a retry loop with cleanup is a simple pattern that prevents frustration.

**CSS variables are powerful but demand precision.** A variable only works if the element using it is a descendant of the element defining it — and only if the consuming property actually references the variable. Hardcoded values break the chain. Attribute selectors are strict: `[data-universe="exploded"]` on the parent doesn't style the child. Keep attributes on the elements that need them.
